== 应用领域扩展（ADE）

应用领域扩展（ADE）是针对特定应用或领域的城市GML概念模型（CM）的正式和系统性扩展。ADE以UML概念模型的形式表示。领域数据被映射到一组额外的类、属性和关系。ADE可以使用城市GML CM中的元素来派生特定于应用的子类、注入额外的属性、将应用数据与预定义的城市GML内容关联，或者为属性定义值域。

ADE机制允许以结构良好且系统化的方式将特定于应用的信息与城市GML CM对齐。通过这种方式，城市GML可以在满足应用的信息需求的同时，保留其概念和语义结构。此外，与通用城市对象和属性不同，应用数据可以针对ADE的正式定义进行验证，以确保语义互操作性。

城市GML标准的早期版本仅在XML模式编码层面定义了ADE机制。随着城市GML 3.0的推出，ADE成为概念层面的平台无关模型，可以映射到多个不同的目标编码。

ADE在实践中已成功实施，并基于城市GML标准启用了一系列广泛的应用和用例。关于现有ADE的概述和讨论，请参阅[<<Biljecki2018>>]。

=== ADE的一般规则

ADE应按照通用要素模型以及<xref:iso19109[ISO 19109]中指定的创建应用模式的UML规则来定义为UML概念模型，并且应遵循xref:iso19103[ISO 19103]中指定的使用UML建模地理信息的规则和约束。在城市GML概念模型中使用的UML符号和构造型（<<uml_notation_section>>）也应应用于ADE中相应的模型元素。

每个ADE应组织成一个或多个具有全局唯一命名空间的UML包，并包含ADE定义的所有UML模型元素。ADE还可以导入并使用外部概念UML模型（例如城市GML模块或ISO 19100系列国际标准的标准化模式）中预定义的类。

=== 定义新的ADE模型元素

根据ISO 19109，地理空间信息的主要视图和应用模式的核心元素是__要素__。因此，ADE通常通过定义适用于应用领域的新的要素类型以及额外的内容（如对象类型、数据类型、代码列表和枚举）来扩展城市GML。

ADE中的每个要素类型都应直接或间接从城市GML根要素类型__Core::AbstractFeature__派生，或者根据其类型和要素，从更合适的子类派生。根据城市GML的一般空间概念，表示空间或空间边界的要素应直接或间接从__Core::AbstractSpace__或__Core::AbstractSpaceBoundary__派生。代表顶级要素类型的UML类应使用__&#171;TopLevelFeatureType&#187;__构造型。

与要素类型不同，对象类型和数据类型不要求从预定义的城市GML类派生，除非另有明确说明。

ADE类可以拥有任意数量的属性和关联，此外还有从其父类继承的属性。属性可以使用简单或复杂的数据类型进行建模。为了确保语义互操作性，应尽可能使用城市GML或ISO 19100系列国际标准的标准化模式中的预定义类型。这包括但不限于ISO/TS 19103中的基本类型、ISO 10107中的几何和拓扑对象以及ISO 19108中的时间几何和拓扑对象。

如果无法使用预定义类型，ADE可以定义自己的数据类型，或者从外部概念模型导入数据类型。这明确包括定义ISO 19107未提供的新几何类型的可能性。然而，ADE的设计者应注意，软件可能无法正确识别和使用这些几何类型。

捕获具有几何形状的真实世界要素的要素类型应直接或间接从__Core::AbstractSpace__或__Core::AbstractSpaceBoundary__派生。通过这种方式，城市GML预定义的空间属性和相关的LOD概念被继承并可用于该要素类型。然而，如果这些基类不合适或者缺少表示要素所需的空间属性，ADE可以定义新的额外空间属性。如果这样的空间属性属于预定义的LOD之一，那么属性名称应以“lod__X__”为前缀，其中__X__应替换为0到3之间的整数值，表示目标LOD。这使得软件能够推导出几何的LOD。

应使用正式语言（如对象约束语言OCL）来表达模型元素的约束。ADE指定了约束的应用方式。然而，按照城市GML概念模型，约束至少应在__Core::AbstractSpace__的ADE子类上表达，以限制可用于建模空间对象边界的/Core::AbstractSpaceBoundary/实例的类型。

=== 为城市GML要素类型添加额外的ADE属性

如果预定义的城市GML要素类型缺少特定应用所需的属性，城市GML 2.0中的可行解决方案是从城市GML类派生新的ADE要素类型子类，并将属性添加到该子类中。虽然从概念上讲是清晰的，但这种方法也存在缺点。如果多个ADE需要为同一个城市GML要素类型添加额外属性，这将导致该要素类型在不同ADE命名空间中有许多子类。关于同一真实世界要素的信息可能会分散在不同要素类的各个实例中，使得软件难以使用要素数据。

因此，城市GML 3.0提供了一种方法，无需子类化即可使用ADE域中的额外属性增强预定义的城市GML要素类型。每个城市GML要素类型都有一个名为“adeOf__FeatureTypeName__”的扩展属性，类型为“__ADEOfFeatureTypeName__”，其中__FeatureTypeName__被定义该属性的类名替换。例如，__Building::Building__类提供了名为__adeOfBuilding__的属性，类型为__Building::ADEOfBuilding__。这些扩展属性可以出现零次或多次，属性类型被定义为抽象且空的数据类型。

如果ADE为特定城市GML要素类型添加额外的ADE属性，ADE应为与要素类关联的相应抽象数据类型创建子类。该子类也应使用__&#171;DataType&#187;__构造型定义为数据类型。然后将额外的特定于应用的属性和关联建模为ADE子类的属性。这可能包括但不限于具有简单或复杂数据类型的属性、空间属性或与ADE或外部模型（如城市GML）中的其他对象和要素类型的关联。

预定义的“_ADEOfFeatureTypeName_”数据类型被称为“钩子”，因为它们被用作附加特定于应用的属性的ADE子类层次结构的头部。在ADE中为特定城市GML要素类型子类化“钩子”时，子类中定义的属性可用于该要素类型以及所有直接或间接派生的要素类型，包括在相同或另一个ADE中定义的要素类型。

多个不同的ADE可以使用“钩子”机制为同一个城市GML要素类型定义额外的ADE属性。由于“adeOf__FeatureTypeName__”属性可以多次出现，因此可以在编码中将各种ADE属性作为同一个城市GML要素实例的一部分进行交换。因此，软件可以轻松使用默认的城市GML要素数据以及来自不同ADE的额外属性。

应用程序或服务在使用编码的城市GML模型时，可能会忽略来自未知或不受支持的ADE的内容。

ADE的设计者应尽可能使用这种“钩子”机制，而不是子类化城市GML要素类型。如果ADE必须启用其他ADE增强其自己的要素类型（所谓的ADE的ADE），那么它应按照城市GML概念模型中相同的模式和命名概念实现其要素类型的“钩子”。

=== ADE的编码

本文档仅涉及ADE的概念建模。将概念ADE模型映射到目标编码的规则和约束预计将在相应的城市GML编码标准中定义。如果得到支持，ADE可以在符合相应城市GML编码标准的情况下提供额外的映射规则和约束。
